# TDD Guard — How It Works

## For Humans

The TDD Guard prevents the AI agent from writing code without tests. It's a state machine enforced by Claude Code hooks that **physically blocks** file edits unless the agent follows the test-first cycle.

When the `implement-2-features` or `implement-2-features-1-page` skill is active, **every** Edit and Write tool call passes through a guard script. The guard checks a log file to determine what the agent is allowed to do right now.

### The Cycle

```
     ┌─────────────────────────────────────────────────────────────┐
     │                                                             │
     ▼                                                             │
 ┌─────────┐   declare RED   ┌───────────────┐   run tests    ┌───┴──┐
 │ initial  │ ──────────────► │ writing_tests │ ──(FAIL)─────► │ red  │
 │          │                 │               │                │      │
 │ nothing  │                 │ can edit TEST │                │ can  │
 │ editable │                 │ files only    │                │ edit │
 └─────────┘                  └───────────────┘                │ noth.│
     ▲                                                         └──┬───┘
     │                                                            │
     │          ┌────────────────────┐   declare GREEN            │
     │          │ making_tests_pass  │ ◄──────────────────────────┘
     │          │                    │
     │          │ can edit DECLARED  │
     └──(PASS)──│ prod files only   │
    run tests   └────────────────────┘
```

### What the agent experiences

1. **Agent tries to create a test file** → BLOCKED. Guard says: "Declare RED first."
2. **Agent declares RED** (names the test file and what it expects to fail) → Now test files are editable.
3. **Agent writes the test, runs it** → Test fails as expected → State moves to `red`.
4. **Agent tries to edit a page component** → BLOCKED. Guard says: "Declare GREEN first."
5. **Agent declares GREEN** (lists exactly which prod files it will change) → Those files become editable.
6. **Agent implements the feature, runs tests** → Tests pass → State resets to `initial`.
7. **Repeat** for the next feature.

### What can't be cheated

| Cheat attempt | What happens |
|---|---|
| Skip tests, edit prod file directly | BLOCKED — state is `initial`, no prod edits allowed |
| Write tests but never run them | Can't reach `red` state — no test failure recorded |
| Claim tests passed without running | Post-bash hook only records actual `pnpm run e2e` results |
| Edit a file not declared in GREEN | BLOCKED — file not in allowlist |
| Delegate to a subagent to bypass | Subagent inherits the same hooks from skill frontmatter |
| Delete the TDD log to reset | Fresh log = `initial` state = still blocked |

### When it activates

The hooks are defined in the skill's YAML frontmatter. They activate **automatically** when `/implement-2-features` or `/implement-2-features-1-page` is invoked, and deactivate when the skill finishes. No manual setup, no flag files. Other skills and general work are completely unaffected.

### Files that bypass the guard

Not everything needs TDD. These file types are always editable:

- `postxl-schema.json` (schema changes trigger regeneration, not direct code)
- `frontend/src/routeTree.gen.ts` (auto-generated by TanStack Router)
- `*.md`, `*.json`, `*.css` (docs, config, styles)
- `e2e/fixtures/*` (test infrastructure, not test specs)
- `scripts/*`, `docker/*`, `.claude/*` (tooling)

### Startup behavior

When the skill starts, no TDD log exists yet. State is `initial` — meaning nothing is editable. The agent's first action that touches a test or prod file will be blocked, and the error message guides it to declare RED. This is by design: the guard is always on, the agent learns the workflow from the error messages.

**Baseline check (Step 0):** Before any TDD work begins, the orchestrator runs the existing e2e test suite. If tests fail, the agent must stop — you don't start new work on a broken baseline. If no e2e tests exist yet, the suite passes trivially and work can begin.
