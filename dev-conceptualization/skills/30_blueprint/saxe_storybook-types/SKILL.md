\_concept/3_blueprint/3_datamodel/seed.json — seed data for type validation
\_concept/2_experience/4_storybook/src/types/index.ts — current mocked types to replace
\_concept/2_experience/4_storybook/src/components/** — custom components (may need type fixes)
\_concept/2_experience/4_storybook/src/pages/** — page compositions (may need type fixes)
\_concept/2_experience/4_storybook/src/stories/\*\* — stories with inline seed data (may need type fixes)

\_concept/2_experience/4_storybook/src/types/ — generated + UI-only types (replaces mocked barrel)
\_concept/2_experience/4_storybook/src/components/** — type fix-ups if needed
\_concept/2_experience/4_storybook/src/pages/** — type fix-ups if needed
\_concept/2_experience/4_storybook/src/stories/\*\* — type fix-ups if needed

# ── Workflow ──────────────────────────────────────────────────────────

STEP 1: Read context

- Read postxl-schema.json to understand models, fields, enums, and relations
- Read the current mocked types in src/types/index.ts
- Inventory all type imports across src/components/, src/pages/, and src/stories/
- Classify each mocked type as:
  - **schema-backed**: has a corresponding model or enum in postxl-schema.json (will be replaced)
  - **UI-only**: no schema counterpart, exists purely for rendering (will be preserved)

STEP 2: Generate types from schema
RUN pxl types \_concept/3_blueprint/3_datamodel/postxl-schema.json --output \_concept/2_experience/4_storybook/src/types

- This generates TypeScript interfaces and enums from the PostXL schema
- Output goes directly into the Storybook's src/types/ directory
  IF pxl types fails
  - Read error output, fix schema issues or command flags, retry

STEP 3: Reconcile types

- Identify which mocked types were replaced by generated types
- Collect UI-only types that `pxl types` did not generate (e.g., NavItem, ChatMessageData, LogEntry, FileTreeNode)
- Write UI-only types to src/types/ui.ts:
  ```typescript
  // UI-only types — not backed by a data model entity.
  // These exist purely for Storybook rendering and will NOT be
  // generated by pxl types.
  ```
- Update src/types/index.ts as a barrel that re-exports everything:
  ```typescript
  // Schema-generated types
  export * from './generated';
  // UI-only types (not in the data model)
  export * from './ui';
  ```
- Adjust the generated file's export name if needed to avoid conflicts with ui.ts exports

STEP 4: Fix compilation errors
RUN cd \_concept/2_experience/4_storybook && pnpm tsc --noEmit 2>&1
IF there are type errors - For each error, determine the cause:
a. **Missing field**: generated type has more required fields than the mocked version
→ add the missing fields to inline seed data objects in stories
b. **Renamed field**: schema uses a different field name than the mock
→ update component props and story data to match the schema field name
c. **Type mismatch**: field type changed (e.g., string enum → union type)
→ update usage to match the generated type
d. **Removed type**: a mocked type was deleted but is still imported
→ either add it to ui.ts (if UI-only) or map to the correct generated type - Fix errors and re-run `pnpm tsc --noEmit`
UNTIL no type errors remain

STEP 5: Verify Storybook build
RUN cd \_concept/2_experience/4_storybook && pnpm run build
IF build fails - Fix errors (may be runtime issues not caught by tsc) - Re-run build
UNTIL build succeeds
