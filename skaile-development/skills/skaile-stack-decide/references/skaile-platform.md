# Skaile Platform -- Agent Context Pack

A condensed reference for AI agents reasoning about whether an app idea
can be implemented **directly inside the Skaile platform** -- via skills,
flows, MCP servers, connectors, custom AAP components, or platform
extensions -- instead of (or before) building a standalone app.

Companion docs:

- `skaile-agent-framework.md` -- the runtime stack
- `skaile-ecosystem.md` -- the whole skaile-dev ecosystem
- `postxl-framework.md` -- PostXL (the generator that produces the platform)

This file zooms into the platform: what it already provides, what
extension surfaces it exposes, and how to map an app idea to one of them.

---

## 1. What the Platform Is

The Skaile platform (`platform/`) is a multi-tenant, browser-based AI
execution environment. End users (business users, IT admins) work in
**projects** that contain **sessions**. Each session runs in a Docker
container with an agent inside. The agent has access to skills, flows,
MCP servers, mounts, and connectors -- all configured per project /
session via a `skaile.yaml` that the platform manages.

Stack: NestJS + Fastify + Prisma + tRPC + PostgreSQL + Keycloak; React
19 + TanStack + Tailwind 4; vm-agent Docker containers + WebSocket IPC.
Generated by PostXL (`postxl-schema.json` is the source of truth).

The platform is the answer to: "I want our company to use AI agents
against our data, with governance, multi-tenancy, audit, and human
review." Many app ideas you might naively prototype as a standalone Nuxt
app (Tier 3) can be implemented as a **set of skills + a flow + a few
custom components** inside the platform, with zero infrastructure,
auth, or persistence work.

---

## 2. The "Should I Build This in the Platform?" Decision

The platform is the right home for an app idea when several of these are
true:

- The user is a non-developer (business user, analyst, knowledge worker).
- The work involves data the company already owns (SharePoint, Postgres,
  S3, GitHub, Slack, ...) and would benefit from governed access.
- The interaction is conversational + artifact-oriented (chat with files
  produced and reviewed).
- There are multi-step workflows with human approval gates between
  steps.
- You want multi-tenancy (orgs, teams, per-user credentials) without
  building it.
- You want voice IO, mentions, reactions, real-time collaboration without
  building them.
- The user shouldn't have to install anything -- a browser is enough.

The platform is **not** the right home when:

- You need a custom UI shell that doesn't fit the chat + artifacts +
  navigator layout (build a forge app instead).
- You need to embed agents inside a third-party host (Mattermost bot,
  CLI tool, mobile app -- look at the forge ladder).
- You need very tight latency or non-conversational workflows (e.g.
  per-keystroke autocomplete).

A useful rule of thumb: **start in the platform, graduate to a forge app
only if you hit a wall**. The platform's extension surface is large.

## 3. The Five Layers (mental model)

Stack the layers from least to most invasive. Each row is a question:
"Can I implement my idea by stopping here?"

| Layer | Mechanism | Invasiveness | Where it lives |
|---|---|---|---|
| 1. **Skills + flows** | Author SKILL.md / flow.yaml; install via the asset catalog | None (data only) | `ai-assets/`, `ai-assets-skaileup/`, or your org's catalog source |
| 2. **MCP servers + connectors** | Declare in `skaile.yaml`; provision credentials via the platform | Config | `ai-assets-skaile/connectors/`, MCP registry, ProviderLink |
| 3. **Custom AAP components** | Web component or iframe registered by the agent at runtime | Frontend asset | Hosted by your org or platform; agent references via URL |
| 4. **App actions** | Declare commands the agent can invoke through the command palette | Frontend wiring | Frontend `*.actions.tsx` files |
| 5. **Platform extensions** | New backend libs, Prisma models, tRPC routes, frontend pages | Code change in `platform/` | Direct edit; PostXL custom blocks |

Most app ideas should land at layer 1 or 2. Layer 3-4 cover the cases
where you need richer UI than chat. Layer 5 is for new product surfaces.

## 4. Layer 1 -- Skills and Flows (the default starting point)

### Skill = unit of agent capability

A `SKILL.md` file with YAML frontmatter and markdown instructions. The
agent reads and follows it at execution time. No deployment, no compile.

What you get for free when you ship a skill:

- It appears in the workspace UI sidebar as something the user can run
- The platform asks for any declared `user_inputs.dialog` fields before
  starting
- The skill is added to `skaile.yaml` per project or session
- Mid-session add/remove via the UI is supported (`configure` message
  triggers a `skills_changed` event without losing chat history)
- The agent discovers the skill natively (Claude SDK `.claude/skills/`)
- Validators (`validator.py`) can be run via the lab to test skill outputs

### Flow = orchestrated multi-step skill pipeline

A YAML/JSON definition with skill nodes and edges (dependencies). State
per node: `available -> running -> awaiting_approval | awaiting_input ->
complete | failed | skipped`.

What you get for free when you ship a flow:

- A flow execution panel in the right sidebar (start, approve, provide
  input, retry, cancel buttons; toggle autonomous mode)
- Persistence in the `FlowExecution` Prisma table -- snapshots every
  `state_changed` event, surviving hibernation/wake
- Chat breadcrumbs (collapsible cards rendered inline with messages
  whenever a flow node transitions)
- Full tRPC mutation surface
  (`workspace.startFlow / approveFlowNode / provideFlowInput /
  cancelFlow / setFlowAutonomousMode / retryFlowNode`)
- Auto-rehydration on container wake (active flows continue where they
  left off without a turn kick)
- Approval/input gates with `decidedBy` / `providedBy` audit fields
- Platform decides when to hide flows whose required skills are missing
  (`status: complete | degraded | unavailable`)

### When to stop here

If your idea reduces to "an agent should do steps A, B, C with optional
human approval between them, using existing data sources" -- you are
done after writing skills, a flow, and (optionally) authoring a domain
catalog package that bundles them.

### Catalog publishing

Skills and flows reach the platform through **catalog packages**.

```yaml
# ai-assets/catalogs/my-domain.catalog.yaml
name: "My Domain"
version: "1.0.0"
description: "..."
include:
  - "my-domain/skills/**"
  - "my-domain/flows/**"
suggested_taxonomy:
  - label: "Research"
    assets: ["my-domain/skills/research-*"]
```

CI compiles the manifest into `<name>.catalog.json`. The platform's
**asset catalog** aggregates compiled catalogs from connected sources
(git, local, remote). Org admins set source priority -- a higher-priority
source can shadow a lower one's skill of the same name. IT can
configure org-specific taxonomy and per-role permissions. Users browse
and add skills/flows from the workspace UI; selections persist into
`skaile.yaml`.

Reference doc: `platform/docs/asset-catalog-architecture.md`.

## 5. Layer 2 -- MCP Servers, Mounts, Connectors

The agent reaches data and external services through three mechanisms,
all declared in `skaile.yaml` and managed by the platform.

### Mounts (filesystem-bound)

The container sees mounted paths under `/skaile/.mounts/<id>/`. Built-in
drivers in `ai-assets-skaile/mounts/`:

| Driver | Use for |
|---|---|
| `local` | Bind-mount a host directory |
| `git` | Clone + sync a git repo (with session lifecycle hooks: branch on session create, merge on close) |
| `s3` | S3 bucket sync |
| `webdav` | WebDAV sync |
| `sharepoint` | SharePoint delta sync |

The platform's project source type (`Git | SharePoint | LocalFolder |
Empty`) drives which mount is used for the project's primary workspace.
Additional mounts can be declared per skaile.yaml.

### Connectors (tool-accessed backends)

The agent calls `connector_read`, `connector_write`, `{id}__query`,
etc. Built-in adapters in `ai-assets-skaile/connectors/`:

Data: `memory`, `postgres`, `redis`, `sqlite`, `xstate`, `xstate-store`,
`yjs`, `gmail`, `minio`, `mattermost`. Service: `devserver`,
`static-server`, `tunnel`, `deploy`.

Each adapter has a `CONNECTOR.md` declaring npm peer deps, configuration
fields, and a health-check operation. The platform reads these to
auto-install peers and validate config.

### MCP servers

Declared in `skaile.yaml` `mcp_servers:`. The Claude SDK driver passes
them as `AgentConfig.mcpServers`; tool calls become native `tool_use`.
For other drivers, MCP tools fall back to the `skaile res` CLI.

```yaml
mcp_servers:
  - id: github
    command: npx
    args: [-y, "@modelcontextprotocol/server-github"]
    env:
      GITHUB_TOKEN: env:GITHUB_TOKEN
```

Or catalog-resolved: `dependencies: [mcp:github]`.

### When to add what

| Need | Mechanism |
|---|---|
| Read or edit project files | Mount (the project source type already provides one) |
| Add a second filesystem source (extra git repo, S3 bucket, SharePoint site) | Additional `mounts:` entry |
| Query a SQL/KV/state-machine backend | Connector |
| Real-time collaborative state agents and humans share | `yjs` connector (CRDT) |
| Turn-based workflow state machine | `xstate` connector |
| External SaaS the agent calls (GitHub, Slack, Jira, Salesforce...) | MCP server (preferred) or new connector adapter (if no MCP exists) |
| In-memory ephemeral KV during a session | `memory` connector |
| Spawn a dev server / static file server / tunnel / deploy | Service connector (`devserver`, `static-server`, `tunnel`, `deploy`) |

## 6. Layer 2 -- Auth, Credentials, Policy (the governance machinery)

This is what makes the platform suitable for real enterprises and what
you almost certainly do not want to build yourself.

### ProviderLink + per-user / service-account credentials

All external provider connections are unified through a `ProviderLink`
in the `Config` schema. Two auth modes per link:

| Mode | What it stores | When to use |
|---|---|---|
| **UserDelegation** | Per-user `UserProviderCredential` (OAuth token, PAT, ...) | Systems where user-level access matters (SharePoint, email, personal CRM) |
| **ServiceAccount** | Per-link `ServiceAccountCredential` (encrypted blob or vault ref) | Shared resources, multi-user sessions, systems without user OAuth (e.g. S3) |

Credential types: `OAuthToken`, `PAT`, `KeyPair`, `KeyFile`, `BasicAuth`,
`VaultRef`. All encrypted as AES-256-GCM JSON blobs.

### Per-session connector access mode

Each session has a `connectorAccessMode`:

| Mode | Default for | Credential source |
|---|---|---|
| `OwnerDelegation` | Single-user sessions | Session owner's `UserProviderCredential` |
| `ServiceAccount` | Sessions with multiple participants | Org's `ServiceAccountCredential` |
| `SharedDelegation` | Owner override (acknowledged risk) | Owner's credential shared to all participants |

Adding a participant to a session may auto-switch the mode (or block
the invite if no service account is configured).

### Skill-level connector policy

Each skill can declare what connectors it needs and with what
permissions:

```yaml
connectors:
  sharepoint:
    mode: user-delegation
    scopes: [Files.Read]
    paths: ["/sites/research/*"]
  github:
    mode: service-account
    account: skaile-bot
    scopes: [repo:read]
  slack:
    mode: service-account
    scopes: [chat:write]
    channels: ["#research-updates"]
```

IT reviews the policy when approving a skill template. The policy
engine is **deterministic code** -- it cannot be prompt-injected.

### Defense-in-depth for prompt injection

Six layers documented in `platform/docs/integration_architecture.md`:

1. Structural separation of prompt vs. content
2. Policy engine (deterministic; the strongest layer)
3. Input scanning (pattern match for known injection signatures)
4. Approval gates (human-in-the-loop for writes)
5. Network segmentation (per-session security level: Open / Standard / Restricted / Isolated)
6. Monitoring and anomaly detection

Audit logs always record the user who triggered each call, regardless of
auth mode.

### Secrets backends

Pluggable -- the platform adapts to the customer's vault:

| Deployment | Backend |
|---|---|
| On-prem | Customer vault (HashiCorp Vault, CyberArk, ...) |
| AWS | Secrets Manager |
| Azure | Key Vault |
| GCP | Secret Manager |
| Dev / small | Env vars or encrypted config |

## 7. Sessions, Containers, Workspaces

Internal model the agent sees:

```
Organization -> Project -> Session -> Conversation -> Message
                            |
                            +-- workspace (isolated working copy)
                            +-- SessionContainer (ephemeral, many over lifetime)
                            +-- SkaileConfig (config: agent config, mounts, connectors)
                            +-- mounts/connectors (per session)
```

Project source type drives session isolation:

| Source | Main session | Branch session | Close behaviour |
|---|---|---|---|
| Git | Main branch worktree | New branch + worktree | Merge to main via `MountManager.closeAll()` |
| SharePoint | Delta-synced local copy | Copy-on-write from main | Driver sync-back |
| LocalFolder | Bind-mount of host dir | Copied subdirectory | Copy back to main |
| Empty | Empty workspace | Empty workspace | No sync |

Lifecycle: `Provisioning -> Running -> (idle) -> Hibernating ->
Hibernated -> (user reconnect) -> Waking -> Running -> Closing -> Closed`.

Hibernate flow:

1. `MountManager.hibernateAll()` + `ConnectorManager.hibernateAll()`
   (git: commit + push; others: driver-specific flush)
2. Persist checkpoint (`commitSha` etc.) on `Session`
3. Snapshot `sessionStoreSnapshot` (presence, UI state, AAP component
   state, reactive entity store)
4. Disconnect gateway, stop container

Wake flow:

1. Spin up new vm-agent in the session's directory
2. Send `configure` with agent config from `SkaileConfig`, shared
   stores, and `activeFlows` (active `FlowExecution` rows resolved
   against the catalog and rehydrated via `orchestrator.hydrate(state)`)
3. Send context restoration prompt (system prompt + skill defs +
   conversation history capped at `contextRestoreLimit` + orientation
   "session resumed after hibernation")
4. Frontend shows history; user continues

Idle detection: `AgentEventService` updates `lastActivityAt` on every
inbound user message and outbound agent event, plus recent workspace
file writes count as activity. Periodic sweep (every ~60s) hibernates
sessions past `session.idleTimeoutMinutes`.

### Configurable platform defaults

| Key | Default |
|---|---|
| `session.idleTimeoutMinutes` | 15 |
| `session.maxHibernationDays` | 7 |
| `session.contextRestoreLimit` | null (all messages) |
| `session.maxActiveSessions` | 10 (per org) |

### Scoped sessions

A session can be scoped to a subfolder of the project workspace via
`scopeRootPath` (now in mount options). Container mount isolation
prevents accidental cross-session writes. Reference:
`platform/docs/scoped-sessions.md`.

## 8. Layer 3 -- Custom AAP Components (Agent App Protocol)

When chat + artifacts + flow panel is not enough, the agent can render
custom UI in four targets:

| Target | Where it renders |
|---|---|
| `chat` | Inside the message stream |
| `preview` | Right-pane preview panel |
| `modal` | Overlay modal |
| `input-extension` | Composer area (extends the text input UI) |

### Component types

```ts
type ComponentSource =
  | { kind: "web-component"; url: string; tagName: string }
  | { kind: "iframe"; url: string }
```

### Lifecycle

1. Agent emits `component_register` once per turn with a manifest
   `{ schema, interactions, streaming, source, targets, fallback,
   messageType }`.
2. Gateway caches the manifest per container; replays it to new
   subscribers.
3. Agent emits `[CUSTOM:type:json]` markers (or `CustomMessageEvent`
   directly).
4. Frontend resolves via native renderers -> matching `messageType`
   component -> fallback markdown template -> plain text.
5. Component state with `persistState: true` is snapshotted into
   `Session.session_store_snapshot` at hibernate and restored on wake.

### When to use

| Need | Approach |
|---|---|
| Render a chart, map, table, custom diagram in chat | AAP web component, target `chat` |
| Show a complex artifact preview (Excel, slides, generated app) | AAP component or iframe, target `preview` |
| Quick interactive form (multi-field input the chat composer can't express) | AAP component, target `input-extension` |
| Modal dialog requesting structured input | AAP component, target `modal` |

You author components anywhere (your own CDN, a static-server connector,
the platform's static assets). The agent registers them; the frontend
loads them with error boundaries.

## 9. Layer 4 -- App Actions (Command Palette)

Every user-facing action in the frontend is registered as a **command
palette action**. This makes it discoverable via Cmd+K **and** invokable
by the built-in AI agent.

Three tiers:

| Tier | Scope | Registration |
|---|---|---|
| Global | Always available | `registerGlobalActions()` |
| Page-scoped | While page is mounted | `registerComponentActions()` (deregister in useEffect cleanup) |
| Component-scoped | While component is mounted | Same as page-scoped |

Visibility:

| Value | Cmd+K | AI agent | Use for |
|---|---|---|---|
| `visible` | Yes | Yes | Default |
| `llm` | No | Yes | Internal/debug actions the AI can use but users don't see |
| `user` | Yes | No | Actions requiring browser interaction (OAuth, sign out) |

When you want the agent to be able to do something the user can already
do in the UI, the answer is "register an app action with `llm`
visibility". When the agent should be able to invoke an arbitrary
backend capability with confirmation, declare it as `AppActionEvent` --
the frontend renders the call, executes it, and returns
`AppActionResultCommand`.

Reference: `platform/CLAUDE.md` "Frontend Action Pattern", and
`agent-framework/types/src/ui-manifest.ts:73-94`.

## 10. Layer 5 -- Backend Libraries (when you need a new product surface)

When skill + connector + AAP component is not enough -- e.g. you need a
new entity, new tRPC routes, new frontend pages -- you reach into
`platform/backend/libs/`.

### Notable libraries already there

| Library | Role |
|---|---|
| `session` | Session workspace lifecycle: hibernate/wake, idle detection, context restoration |
| `agent-gateway` | WebSocket bridge to vm-agent containers; connection pool; resource and state caches; `state_changed` write-through for flows |
| `ai` / `ai-provider` | Provider-agnostic LLM dispatch |
| `skill-registry` | Catalog aggregation, taxonomy, permissions |
| `deployment-manager` | Phase-based pipeline engine (PostXL generation runs) |
| `preview-manager` / `preview-proxy` | Generated-app preview lifecycle (start, refresh, stop; container + proxy) |
| `voice` | STT (Deepgram) + TTS (ElevenLabs) WebSocket protocol |
| `git-provider` / `sharepoint-provider` / `nextcloud-provider` / `s3` | Project source backends |
| `credential` / `secrets-provider` | Multi-provider secret resolution |
| `artifact-manager` / `excel-extraction` / `excel-io` / `xlport` | Document handling |
| `import` / `mail` / `upload` | Ingest / outbound mail / upload |
| `capabilities` | Declared agent capabilities discovery |
| `e2e` | E2E harness helpers |

### Adding a new feature: PostXL workflow

1. Edit `postxl-schema.json` (source of truth) -- add the model.
2. `bun pxl generate` to regenerate Prisma + tRPC + types + actions.
3. Add `bunx prisma migrate dev` for the new table.
4. Extend generated files via `// @custom-start:<name>` ... `// @custom-end:<name>`
   markers (preserved across regeneration).
5. Wire frontend pages, hooks, and command palette actions.

**Never** edit `prisma/generated/` directly. **Never** create a barrel
file in `platform/backend/libs/` (breaks NestJS DI). Use direct subpath
imports.

### Useful protocol extensions to know about

Documented in `platform/docs/protocol-extensions.md`. The platform layers
these on top of the base agent-framework messaging:

- **Inline text markers** in agent output (the agent emits these as plain
  text and the platform extracts them):
  - `[REACT:emoji]` or `[REACT:emoji:N]` -> creates a `MessageReaction`
    on the latest user message or message with `seq=N`
  - `[PASS]` -> in multi-user mode, suppresses the agent's turn
  - `[CUSTOM:type:json]` -> renders a custom AAP component (the runner
    buffers across chunks and parses)
- **Mentions** -- frontend parses `@user` and `@group` (`agent`,
  `humans`, `humans_here`, `here`, `all`) tokens; resolved IDs frozen
  at parse time
- **Reply threading** -- `prompt.replyTo` becomes
  `Message.parent_message_id` on persist
- **Reactions** -- per-user emoji, agent reactions use `userId="agent"`
- **Voice** -- `POST /voice/speak` (TTS), `WS /voice/transcribe-stream`
  (Deepgram STT)
- **Snapshot events** -- LLM-compacted conversation summaries; on wake
  only the latest snapshot + newer messages are replayed
- **App actions** -- declared agent capabilities with description, param
  schema, execution location, required roles, confirmation requirement
- **Lifecycle commands** -- `{ type: "lifecycle", phase:
  "hibernate"|"close"|"compact" }` for clean shutdown verification
- **Flow breadcrumbs** -- `flow_breadcrumb` messages rendered as
  collapsible cards inline with chat

## 11. The Decision Tree (which layer does my idea live in?)

Walk top to bottom; stop at the first row that fits.

1. **"The agent should do steps A, B, C in order, possibly with human
   approval, using existing capabilities."** -> Layer 1: write skills +
   a flow + (optionally) a domain catalog package.
2. **"The agent should also be able to read/write data in system X."**
   -> Layer 2: add an MCP server (preferred) or a connector (if no MCP
   exists) or a mount (if it's filesystem-shaped). Configure a
   ProviderLink for credentials.
3. **"The agent's output is richer than text -- a chart, a custom form,
   an interactive widget."** -> Layer 3: ship an AAP component, agent
   emits `[CUSTOM:...]` to invoke it.
4. **"The agent should be able to operate the platform UI itself
   (navigate, open dialogs, run commands the user can run)."** -> Layer
   4: register app actions; agent invokes them via `AppActionEvent`.
5. **"I need a brand-new entity / page / endpoint that doesn't fit the
   project/session model."** -> Layer 5: PostXL schema edit + new lib
   under `platform/backend/libs/`.

For most ideas, layers 1-3 are sufficient.

## 12. Concrete App Idea -> Platform Mapping

| App idea | Likely platform shape |
|---|---|
| "Research assistant for our analyst team using SharePoint + web" | Skills (`research`, `summarize`, `cite`) + flow with approval before publishing + ProviderLink for SharePoint (UserDelegation) + MCP server for web search (Exa / Perplexity) |
| "AI co-pilot to run reports against Postgres + S3" | `postgres` connector + `s3` mount + skills (`schema-explore`, `query-build`, `report-render`) + AAP component for chart rendering in `preview` |
| "Generate slide decks from a brief with brand consistency" | Skills (`brief-extract`, `slide-outline`, `slide-render`) using `skaileup-` patterns + `use-pptx` skill + theme tokens + AAP component for slide preview |
| "Onboard new clients: read intake form, draft proposal, send for review" | Flow with approval gate + `gmail` connector (write requires approval) + form via AAP `input-extension` component + skills for each step |
| "Run our regulatory compliance checks across docs in SharePoint" | `sharepoint` mount + skills (`check-policy-X`, `check-policy-Y`, ...) bundled into a domain + flow with `Restricted` security level + audit log |
| "Multi-user concept design (PM + designer + dev)" | This is what `forge/L5-concept` already does -- consider porting its UX into the platform via skills + flows + AAP components, or use the forge app standalone |
| "Generate an internal app from a brief" | The skaileup pipeline (`grounding -> discovery -> experience -> architecture -> datamodel -> build -> quality`) is the canonical path; the platform's `deployment-manager` orchestrates it |
| "Voice-driven note-taking with action extraction" | Use voice STT/TTS already wired in; skills for `transcribe`, `extract-actions`, `route-to-system`; service-account connectors to push actions into Jira/Asana |
| "Data quality auditor for our warehouse" | `postgres` (or warehouse-specific) connector + skills (`profile-table`, `find-anomalies`, `propose-fixes`) + flow with batched-approval write mode + audit log |
| "Customer support copilot drafting replies in our helpdesk" | Helpdesk MCP or new connector + skills (`triage`, `find-similar`, `draft-reply`, `apply-tone`) + queue-for-approval write mode |

The pattern is repeatable: pick the data sources (mounts / connectors /
MCP), pick or write the skills, optionally compose them into a flow,
add an AAP component if chat output isn't sufficient, and configure
auth / policy through the platform's built-in machinery.

## 13. Authoring Checklist (for an "implement on the platform" idea)

1. **Identify data sources** -> for each: mount / connector / MCP.
2. **Configure ProviderLinks** in the platform admin -- pick UserDelegation
   or ServiceAccount per provider.
3. **Author skills** under `ai-assets/<my-domain>/skills/` (use the
   `skill-builder` skill). Declare `connectors:` in skill frontmatter
   with required mode, scopes, paths.
4. **(Optional) Compose a flow** -- nodes are skill IDs; declare edges
   for ordering; mark approval gates with `awaiting_approval`.
5. **(Optional) Author an agent definition** -- SOUL/RULES/agent.yaml/
   knowledge if you need a specific persona / hard constraints.
6. **Bundle** in a `*.bundle.yaml` and a `*.catalog.yaml` for one-shot
   installation.
7. **Connect the catalog source** in the platform admin UI.
8. **Configure taxonomy and permissions** -- who can see and run what.
9. **(Optional) Ship AAP components** -- web component or iframe; agent
   registers via `component_register`.
10. **(Optional) Register app actions** in frontend if the agent should
    drive the UI.
11. **Test** in a session; iterate; pin a `version:` for stability.

## 14. Anti-Patterns and Gotchas

- **Hardcoding credentials in skills.** Use `auth: env:VAR` /
  `oauth:key` / `op:ref` references. Never embed tokens in SKILL.md.
- **Authoring skills that assume a specific filesystem layout.** Skill
  paths should resolve through mounts (`{{workspace}}/...`); never
  assume a specific host directory.
- **Building a custom subprocess agent host inside the platform.** The
  vm-agent container architecture exists; do not bypass it. Add a new
  `IContainerManager` if you need a different hosting backend (Docker,
  Nix, k8s).
- **Editing generated Prisma files directly.** Edit
  `postxl-schema.json` and run `bun pxl generate`. Use custom blocks for
  extensions.
- **Creating a barrel `index.ts` in `platform/backend/libs/`.** Breaks
  NestJS DI module boundaries. Use direct subpath imports.
- **Running Biome on `platform/`.** Platform uses Prettier + ESLint
  exclusively. `platform/biome.json` disables Biome as a safeguard.
- **Skipping the audit log.** All connector calls go through the
  Connector Runtime which audits with auth mode + identity. Don't
  bypass it.
- **Bypassing the approval gate for writes.** The default for any
  external write is "queue for approval". Auto-approve only for low-risk
  internal actions; "blocked" is a valid mode for read-only skills.
- **Treating prompt injection as an LLM concern.** It is a *systems*
  concern. The deterministic policy engine (Layer 2 of defense in
  depth) is what stops most attacks; design skill connector policies
  tightly.
- **Forgetting flow rehydration.** If your flow holds long-lived state,
  remember it survives hibernate but only if the catalog still resolves
  the flow def at wake. Pin versions; do not break flow definitions
  while sessions are active.

## 16. The Platform Mental Model in One Paragraph

The Skaile platform is a multi-tenant browser app that hosts AI agents
in per-session Docker containers. Each session has an agent inside, a
workspace (isolated working copy of the project's data, source-type
dependent), and a `skaile.yaml` describing skills, flows, mounts, and
connectors. The agent drives turn-based execution; the platform owns
durable state (Postgres-backed `Project`, `Session`, `Conversation`,
`Message`, `FlowExecution`, `MessageReaction`, ...), governance
(`ProviderLink` + per-user / service-account credentials, deterministic
policy engine, audit log), and the UI (chat + artifacts + flow panel +
asset catalog browser, AAP components, voice IO, command palette). To
implement an app idea here, walk five layers: skills + flows ->
mounts/connectors/MCP -> AAP components -> app actions -> backend
extensions, and stop at the first layer that fits. Most ideas stop at
layer 1 or 2. The platform's job is to make those layers free.
